# Контекст сайту

Цей файл використовую для фіксації вимог, рішень і проміжного контексту по сайту.

## Нотатки

- Репозиторій був порожній на момент старту.
- Цей файл є робочим джерелом контексту, куди зберігаються всі наступні інструкції.

## Інструкція 001: продукт і MVP ветклініки

### Назва та формат

- Проєкт: веб-застосунок ветклініки.
- Формат: публічний сайт, кабінет клієнта, адмін-панель, окрема зона лікаря.
- Фокус: MVP без перевантаження другорядним функціоналом.

### Загальна мета

Система має дозволяти:

- презентувати клініку та послуги;
- реєструвати й авторизовувати користувачів;
- вести профілі клієнтів і їхніх тварин;
- переглядати історію лікування;
- бачити діагнози, призначення, аналізи, чеки;
- записуватися онлайн через календар;
- адміністратору керувати календарем, записами, клієнтами, тваринами, лікарями, послугами та фінансами;
- лікарю вести медичні записи по прийомах.

### Обов'язковий стек

- Next.js `App Router`
- TypeScript
- PostgreSQL
- Prisma ORM
- Auth.js / NextAuth
- Tailwind CSS
- shadcn/ui

### Технічні принципи

- модульна структура;
- чиста архітектура без хаосу в папках;
- server-side перевірка доступів;
- ролі користувачів;
- зрозумілі типи та інтерфейси;
- масштабованість під подальше розширення.

### Ролі

#### Guest

- бачить лендінг;
- бачить сторінки послуг, лікарів, цін, контактів;
- може перейти до запису;
- може зареєструватися або увійти.

#### Client

Може:

- переглядати і редагувати свій профіль;
- переглядати своїх тварин;
- бачити картки тварин;
- переглядати історію лікування;
- бачити діагнози, призначення, аналізи, вакцинації, чеки;
- створювати онлайн-записи через календар;
- скасовувати або переглядати свої записи;
- бачити найближчі візити і нагадування.

Не може:

- бачити чужі дані;
- редагувати медичні записи;
- бачити внутрішні службові поля клініки.

#### Admin

- бачить дашборд;
- керує календарем записів;
- створює записи вручну;
- редагує, переносить і скасовує записи;
- веде базу клієнтів;
- веде базу тварин;
- керує лікарями;
- керує послугами;
- бачить медичні картки;
- виставляє чеки та оновлює статуси оплат;
- керує графіками лікарів і блокуванням слотів.

#### Doctor

- бачить свій розклад;
- бачить тільки прив'язані до нього записи;
- відкриває картку тварини;
- створює записи візиту;
- додає анамнез;
- додає діагнози;
- додає призначення;
- прикріплює результати аналізів або медичні файли;
- додає рекомендації;
- переглядає історію лікування пацієнта.

#### Superadmin

- має всі можливості `admin`;
- керує ролями;
- змінює глобальні налаштування;
- керує системними параметрами.

### Бізнес-логіка

#### Реєстрація

- реєстрація через email або телефон;
- після реєстрації створюється `User`;
- після реєстрації створюється `OwnerProfile`;
- роль за замовчуванням: `client`.

#### Авторизація

- Auth.js / NextAuth;
- credentials login;
- email/password;
- session-based auth;
- middleware для захисту роутів.

#### Доступи

- клієнт бачить тільки своїх тварин, свої записи, свої чеки, свої меддані;
- doctor бачить тільки пов'язаних із ним пацієнтів і записи;
- admin бачить всю клініку;
- на кожному захищеному маршруті потрібна server-side перевірка доступу.

#### Онлайн-запис

Клієнт:

- обирає тварину;
- обирає послугу;
- обирає лікаря або "будь-який доступний";
- обирає дату;
- бачить доступні часові слоти;
- створює запис.

Система:

- не дозволяє бронювати минулий час;
- не дозволяє бронювати зайняті слоти;
- враховує графік лікаря;
- враховує блокування слотів;
- враховує тривалість послуги;
- перевіряє доступність слота повторно на сервері під час створення запису.

## Поточний статус реалізації

### Що вже реалізовано

#### Архітектура і база

- проєкт побудований на `Next.js App Router`, `TypeScript`, `Tailwind CSS`, `shadcn/ui`, `Prisma`, `Auth.js`;
- піднята Prisma schema для користувачів, клієнтів, тварин, лікарів, послуг, графіків, записів, візитів, діагнозів, призначень, аналізів, вакцинацій, рахунків і нотифікацій;
- є `seed`-дані для базових ролей і демо-сценаріїв.

#### Auth і ролі

- працює credentials login;
- реалізовані ролі `guest`, `client`, `admin`, `doctor`, `superadmin`;
- захищені зони `cabinet`, `admin`, `doctor`;
- є logout-flow для авторизованих користувачів у захищених зонах і на публічній частині.

#### Client flow

- реалізований CRUD тварин;
- працює реальна картка тварини;
- у кабінеті доступні сторінки:
  - записи;
  - візити;
  - призначення;
  - аналізи;
  - рахунки;
- клієнт може скасовувати і переносити власні записи;
- клієнт бачить медичну історію й документи по своїх тваринах.

#### Admin flow

- є адмінський дашборд;
- реалізований `appointments/calendar flow`;
- адмін може:
  - створювати запис вручну;
  - редагувати і переносити запис;
  - змінювати статус;
  - блокувати слоти;
  - фільтрувати по лікарю, даті і послузі;
- `admin calendar` переведений у `day-grid` формат.

#### Doctor flow

- лікар бачить власний розклад;
- відкриває картку пацієнта;
- створює `Visit`;
- додає:
  - `Diagnosis`;
  - `Prescription`;
  - `LabResult`;
  - `Invoice`;
- `doctor schedule` теж переведений у `day-grid`.

#### Uploads

- працює локальний upload flow;
- файли зберігаються через storage layer;
- у doctor flow можна завантажувати вкладення, аналізи й документи;
- клієнт бачить ці файли в кабінеті.

#### Booking

- реалізований покращений booking flow;
- доступний вибір:
  - тварини;
  - послуги;
  - лікаря;
  - режиму `будь-який доступний лікар`;
  - дати;
  - слота;
- є окремий availability layer і API для слотів;
- public booking переведений на `day-grid / slot picker UI`;
- є empty states, тижнева навігація і підказка до найближчого доступного дня.

#### UX / feedback

- підключені toast-нотифікації;
- додані inline success/error banners;
- форми отримали більш послідовний feedback pattern.

### Layout і spacing

- на публічній частині прибраний окремий верхній header;

## Інструкція 002: файли в MVP і модель клініки

### Завантаження файлів у MVP

#### Архітектурна вимога

Потрібно закласти storage-абстракцію для:

- м'яка медична палітра: білий + один акцентний колір;
- фокус на довірі, ясності та читабельності;
- секції на базі card-like блоків;
- чіткі CTA-блоки;
- без візуального шуму.

- використовувати `shadcn/ui` як основу компонентів;
- будувати інтерфейс через cards, buttons, grids;
- responsive layout: desktop-first, але адаптивний;
- уникати перевантаження декором.
